# A2 训练流程改进指导文档

> **适用对象**：实施改进的 agent / 工程师
> **目标项目**：AdoDAS2026 Track A2 baseline（`grouped` 训练流程）
> **依据**：2026-05-26 两次完整 40-epoch 训练日志分析（run 11:45 与 run 19:36）
> **本文档不包含具体代码**：所有改动的具体落点、变量名、文件路径由实施方结合项目代码结构自行确定。

---

## 0. 必读：诊断结论摘要

实施方在动手前必须先理解以下事实，否则会做出错误的修改方向：

1. **不是经典的"训练集过拟合"**：训练 main_loss 仅从 1.87 降到 1.35（下降约 27%），模型并未记住训练集。因此**单纯加 dropout、加 label smoothing、缩小模型规模这类标准抗过拟合手段不是首选方案**。
2. **峰值 QWK 出现在 epoch 7–16**：之后 24 个 epoch 的训练让 QWK 从 ~0.44 单调退化到 ~0.31。当前 40 epoch 的训练长度中约 60% 在做负功。
3. **val_loss 与 QWK 拐点错位**：Run 2 中 val_loss 最低在 epoch 10，而 QWK 峰值在 epoch 7。**以 val_loss 作为 early stop 指标会系统性错过 QWK 最佳点**。
4. **真正驱动 QWK 退化的机制**有三层：
   - **不确定性加权失控**：task_2/task_3 权重从初始 1.0 增长到 5.0 左右，主任务（task_0/task_1）被压到 0.7，主任务梯度被边缘化。
   - **预测分布扁平化**：末期预测分布从健康的 ~75/17/5/0.5 退化为 ~47/45/7/0.3，**类 3 几乎消失**，类 0 严重渗入类 1。QWK 是二次加权，分布扁平化会精准杀掉它。
   - **解码策略不稳定**：训练后期 auto decode 在 argmax 与 expectation 之间反复跳，这是 CORAL 阈值不再可信的症状。

实施方在任何修改后都应**优先关注预测分布是否健康（接近 GT 的 69/23/5/2.5）**，这是比 val_loss 更直接的信号。

## 1. 修改项总览

| 优先级 | 修改项 | 风险 | 预期收益 | 是否阻塞下一项 |
|---|---|---|---|---|
| P0 | Early stopping 指标 + 训练长度 + 学习率调度同步 | 极低 | 节省 60% 训练时间，避免负功 | 是 |
| P1 | 不确定性加权约束 | 中 | 阻止主任务被辅助任务边缘化 | 否 |
| P2 | 辅助任务权重整体下调 | 低 | 让主任务梯度回到主导位置 | 否 |
| P3 | pos_weight 上限收紧 | 低 | 缓解末期预测分布扁平化 | 否 |
| P4 | 核查 QWK 辅助损失是否真正生效 | 低 | 修复潜在的配置失活 bug | 否 |
| P5 | 多 seed / K-fold 评估 | 低 | 区分真实改进与验证集噪声 | 否 |

**实施顺序原则**：P0 必须先做且独立验证，之后 P1–P4 可以单变量逐项验证或合并验证，P5 在前几项确定有效后再做。

## 2. P0：Early stopping、训练长度、学习率调度

### 目标
让训练在 QWK 峰值附近自然停止，不再让后期训练破坏已经达成的最佳模型。

### 修改要点

**(a) 早停监控指标**
- 从 `val_loss` 改为主指标（A2 任务的主指标是 mean QWK）。
- 早停方向：QWK 是越大越好，需要相应调整 mode（与 A1 任务保持一致的方向语义）。

**(b) 早停 patience**
- 从当前的 99（实际等于不早停）降到 5–8 之间。
- 选择依据：从日志看，QWK 一旦掉头，连续 4–6 个 epoch 都难以回到峰值，patience=5 既能容忍小波动，又能及时止损。

**(c) 总 epoch 数**
- 从 40 降到 16–20。
- 选择依据：两次 run 的 QWK 峰值都在 epoch 7–16 之间出现，留少量余量即可。

**(d) 学习率调度周期同步**
- warmup 维持 3 epoch 不变。
- cosine 衰减的总周期需要**同步**调整到新的总 epoch 数。
- 不能只改 epochs 不改 schedule —— 否则会在新的总长度下用偏高的学习率收尾，反而更不稳定。

**(e) Best checkpoint 选择口径**
- 确认保存的 best checkpoint 是按 **QWK** 选择的，而不是按 val_loss。
- 当前代码似乎已经按 QWK 保存（日志里 `>>> New best QWK=...`），但配置项里 `early_stop_metric=val_loss` 与之矛盾。需要核对实际逻辑并统一。

### 验证标准
- 训练时间从 ~1h 缩到 ~25–30min。
- 训练在 epoch 12–18 之间自然停止（不应该跑满 16 个 epoch 还未触发早停，除非曲线异常）。
- 训练终止时记录的 best QWK 应不低于历史 run 的最佳值（~0.44 raw / ~0.46 校准后）。

### 回滚条件
- 如果早停在 epoch 8 之前就触发，说明 patience 太短，调到 8–10 再试。
- 如果连续 3 次 run 的早停 epoch 都在 16+ 且 QWK 仍在上升，把 epochs 上限放回 25 再观察。

## 3. P1：不确定性加权约束

### 目标
阻止 uncertainty weighting 把主任务权重压到 0.7、辅助任务权重抬到 5.0 的失衡状态。

### 背景证据
Detailed losses 显示：

| Epoch | task_0_w (主) | task_1_w (主) | task_2_w (辅) | task_3_w (辅) |
|---|---|---|---|---|
| 1 | 1.00 | 1.00 | 1.00 | 1.00 |
| 10 | 0.74 | 0.71 | 1.97 | 1.68 |
| 25 | 0.69 | 0.65 | 4.27 | 3.41 |
| 40 | 0.71 | 0.67 | 4.97 | 3.95 |

这个走势意味着训练后期主任务梯度被显著压制。

### 修改要点（三选一，按风险从低到高排序）

**方案 A（最保守，推荐先试）**：保留 uncertainty weighting，但**给每个 sigma 加上下限钳制**，避免 sigma 无限增大导致权重失衡。具体钳制范围由实施方根据训练前几个 epoch 的 sigma 自然分布确定，确保不会触及训练初期的合理区间。

**方案 B（中等）**：保留 uncertainty weighting，但**对主任务的 sigma 做 detach** 或**单独施加正则项**，防止主任务 sigma 被推高（被推高意味着主任务被边缘化）。

**方案 C（激进）**：**完全关闭 uncertainty weighting**（`use_uncertainty_weighting: False`），改用固定权重。权重值由实施方根据每个任务 loss 的初始量级估算，保证主任务在总 loss 中的占比不低于 50%。

### 验证标准
观察 detailed losses 中的 task 权重：
- 训练全程主任务权重不应低于 0.85。
- 训练全程任何辅助任务权重不应超过 2.0。
- 若上述任一指标越界，说明约束没生效或不够强。

### 与 P0 的关系
P1 是独立改进，但**应在 P0 完成之后再实施**。原因：P0 缩短训练后，task 权重还没来得及失衡训练就结束了，P1 的边际收益会比在 40 epoch 训练上小。需要先量化 "缩短训练后 P1 还能带来多少额外提升"。

## 4. P2：辅助任务权重整体下调

### 目标
即便不动 uncertainty weighting 机制本身，也通过降低辅助任务的**基础权重**让主任务回到主导位置。

### 修改要点
当前配置中的辅助任务权重清单（按 Run 1 配置）：

| 配置项 | 当前值 | 建议方向 |
|---|---|---|
| `session_loss_weight` | 0.5 | 0.2–0.3 |
| `session_type_loss_weight` | 0.15 | 0.05–0.1 |
| `emotion_dims_weight` | 0.2 | 0.05–0.1 |
| `emotion_cls_weight` | 0.15 | 0.05 或关闭 |
| `au_pred_weight` | 0.1 (Run 1) | 维持关闭 |
| `aux_lupi.phase1` 各 aux 权重 | 0.05–0.2 | 整体砍半 |

**建议方向是单调下调，不是替换值**。实施方应做敏感度实验：先把所有辅助权重 ×0.5，看 QWK 反应；如有提升，再 ×0.5；若有下降则回退。

### 关键判断
Run 2 相对 Run 1 已经关闭了 emotion_cls 和 au_pred，但最终 QWK 几乎相同（0.4554 vs 0.4681 校准后）。这说明**单纯关闭辅助任务收益有限**，更可能的瓶颈在 uncertainty weighting 的动态权重。所以 P2 与 P1 应配合考虑，**不要单独做 P2 就期望大幅提升**。

### 验证标准
- 主任务的 `main_loss` 在训练全程的下降幅度应比当前更大（当前 1.87 → 1.35，目标至少 1.87 → 1.2）。
- 预测分布在中后期 epoch 应更接近 GT 分布（特别是类 3 的比例不应跌到 0%）。

---

## 5. P3：pos_weight 上限收紧

### 目标
缓解末期预测分布扁平化（类 3 消失、类 0/1 互渗）。

### 背景
当前代码对 A2 的 pos_weight 上限是 10.0（按 21 题 × 3 阈值独立计算）。在 uncertainty weighting + 长训练的共同作用下，pos_weight=10 反而成了破坏因素——它鼓励模型对稀有类输出更"激进"的预测，但当主任务梯度被压制后，模型只能用"扁平化预测"来平衡。

### 修改要点
- pos_weight 上限从 10.0 降到 **5.0**（首选）或 **6.0**（保守）。
- 不要改变 pos_weight 的计算公式（`sqrt(n_neg/n_pos)`），只改 clamp 上限。
- 实施完成后**在 P1 已经生效的前提下评估**，否则两个因素混在一起难以归因。

### 验证标准
- 末期预测分布中类 3 的比例不应低于 1%（GT 是 2.5%）。
- 末期预测分布的总体形态应仍接近 GT（即 0/1/2/3 比例约为 70/22/5/2，可有 ±5% 浮动）。

### 注意
若 P3 单独实施后类 3 比例反而上升过多（比如超过 5%），说明上限降得过多，调回 6.0 或 7.0。

---

## 6. P4：核查 QWK 辅助损失生效性

### 目标
当前配置中存在 `use_qwk_aux: 1` 与 `qwk_weight: 0.3`，但日志的 detailed losses 中**找不到对应的损失分量**（如 `aux_qwk_loss`）。需要排查这个配置是否实际生效。

### 排查要点
1. 检查 `use_qwk_aux` 配置项在训练代码中是否被读取并连接到实际的 QWK 可微分损失计算。
2. 检查 `qwk_weight` 是否被乘到了一个真实在反向传播中的 loss 张量上。
3. 若有 QWK aux loss，确认它是否被 uncertainty weighting 也"管理"了 —— 如果是，可能它的权重已经被压到接近 0（参考 P1 的失衡数据）。
4. 在 detailed losses 日志输出中**显式打印 `aux_qwk_loss` 分量**，方便后续追踪。

### 修改要点（若发现未生效）
- 若发现 `use_qwk_aux` 实际上是无效配置：明确实现一个可微分 QWK 近似（如基于 soft confusion matrix 的实现），加入主任务 loss。
- 若已生效但被 uncertainty weighting 压制：让 QWK aux loss 不进入 uncertainty weighting 体系，使用固定权重 0.2–0.5。

### 验证标准
- detailed losses 中应能看到非零的 `aux_qwk_loss` 数值，且整个训练过程中数值应在下降（而非保持为 0 或恒定）。
- 启用 QWK aux 后的 best QWK 应比关闭时高至少 0.005（如果改动正确）。

---

## 7. P5：多 seed / K-fold 评估

### 目标
区分"改进真实有效"与"验证集（仅 600 人）的随机波动"。

### 修改要点
- 在 P0–P4 全部完成后，对**最终配置**用至少 3 个 seed 重跑，比较 QWK 的均值与方差。
- 资源允许时进一步做 5-fold cross-validation（在训练集 4200 人内做切分），这是更稳健但更昂贵的方案。
- 单 seed 提升小于 0.005 的改动不应认为有效（验证集大小决定的 QWK 噪声水平约为这个量级）。

### 验证标准
- 同一配置 3 seeds 的 QWK 标准差应 < 0.01。
- 改进后的 mean QWK 减去 baseline mean QWK 应大于 2 倍标准差，才算统计有效。

## 8. 总体实施与验证流程

### 推荐节奏

```
Baseline run（当前配置）                  ← 已有日志，QWK ≈ 0.44 raw / 0.46 校准
   │
   ▼
[P0] 修改 early stop + epochs + schedule  ← 应得到相近 QWK，但训练时间砍 60%
   │
   ▼
[P1] 加 uncertainty weighting 约束        ← 期望 QWK +0.01 到 +0.03
   │
   ▼
[P2] 调辅助任务权重                       ← 期望 QWK +0.005 到 +0.02
   │
   ▼
[P3] 收紧 pos_weight 上限                 ← 期望 QWK +0.005 到 +0.015
   │
   ▼
[P4] QWK aux loss 核查 / 启用             ← 期望 QWK +0.01 到 +0.02（若原本未生效）
   │
   ▼
[P5] 多 seed 验证最终配置                 ← 确认改进的统计显著性
```

### 每一步的产出要求

每实施一项后，agent 应输出：

1. 修改了哪些配置项 / 代码位置（用相对路径表达，例如 `tasks/a2/default.yaml`、`runner.py:XXX-YYY`）。
2. 训练日志中关键指标的对照：
   - 训练终止 epoch
   - Best QWK（raw 与校准后）
   - 训练终止时的预测分布
   - detailed losses 中各 task 权重的末值
3. 与上一轮相比的差异 delta（QWK、训练时间、预测分布偏移）。

### 红线（任何一项触发立即回滚）

- 训练在 epoch 5 之前就早停。
- Best QWK 比上一轮下降超过 0.01。
- 预测分布末态的类 0 比例超过 90% 或类 1+2+3 之和低于 10%（说明模型退化为常数预测）。
- 训练 loss 出现 NaN 或不收敛迹象（loss 持续上升超过 3 个 epoch）。

## 9. 不要做的修改

以下方向看起来合理，但**基于日志分析不推荐先做**，避免实施方误判优先级：

1. **加大 dropout / 加大 label smoothing / 加大 feature noise**。
   - 理由：train main_loss 没有崩到 0，模型并未严重记住训练集，传统正则收益有限。
2. **降低模型容量 / 减少 TCN 层数**。
   - 理由：同上，目前 ~5.6M 参数对 4200 训练样本不算过分，且现在的瓶颈是任务平衡，不是表达能力。
3. **切换 SSL backbone**。
   - 理由：会引入大量混杂变量，应在训练流程本身稳定后再做特征/backbone 的消融。
4. **直接增大学习率追求快速收敛**。
   - 理由：当前 1e-3 已经触发了 epoch 16 后的退化，更大 lr 只会让退化提前发生。
5. **修改 CORAL 头结构**。
   - 理由：CORAL 阈值漂移是症状不是病因，根因在主任务被边缘化。先稳住训练流程，再视情况考虑头部结构。
